<objective>
Create PinScreen.tsx — a dedicated screen at /pin that handles both first-time PIN creation (CREATE mode) and subsequent PIN verification (VERIFY mode). Register the /pin route in App.tsx. The lock icon on HomeScreen (added in Plan 03) navigates here; this plan defines the destination and its logic.

Rate-limiting: an isVerifying state flag disables digit input while verifyPin() is in-flight, preventing double-tap submissions. After a failed attempt, a 300ms delay restores input — minimal brute-force friction with no lockout.

Loading state: useLiveQuery returns undefined on the first render tick (loading). This is explicitly handled as a loading state distinct from null (no PIN set / CREATE mode).

Purpose: Satisfies PAR-01 (parent section gated behind 4-digit PIN). Enforces the PIN security boundary so /parent is not navigable without authentication.

Output: PinScreen.tsx (CREATE + VERIFY modes, isVerifying guard, loading/null distinction, digit-dot display, hashPin/storePinHash/verifyPin calls), PinScreen.test.tsx (full flow coverage including rate-limit behaviors), /pin route in App.tsx.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → digit buttons | User taps 0–9; only these values enter the digits buffer |
| digits buffer → hashPin() | Raw PIN string crosses into crypto layer — must hash before any DB write |
| hashPin() output → storePinHash() | 64-char hex digest stored in appConfig table — never raw digits |
| /pin → /parent | Navigation only happens after verifyPin() returns true or storePinHash() succeeds |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-01 | Information Disclosure | PinScreen state (digits, firstEntry) | mitigate | Raw digits live only in React useState during entry; never logged, never written to DB; component unmounts after navigation, clearing state |
| T-5-02 | Elevation of Privilege | Direct navigation to /parent bypassing /pin | mitigate | App.tsx routes do not guard /parent at the router level (v1 local-only accepted per PROJECT.md CR-01 note); PinScreen is the only UI path to /parent in the app; HomeScreen lock icon navigates to /pin (Plan 03) |
| T-5-03 | Elevation of Privilege | PIN brute force | mitigate | isVerifying flag blocks input during verification; 300ms delay after failed attempt before input re-enables; no lockout per D-04 (v1 decision — physical device access required, parent controls device) |
| T-5-04 | Tampering | storePinHash called with raw PIN | mitigate | storePinHash() throws at runtime if argument is not 64-char hex (verified in db.ts); always call hashPin() first |
| T-5-05 | Tampering | Non-digit characters entering digits buffer | mitigate | No text input in PinScreen; only digit buttons (0–9) can append to digits buffer; handleDigit guards isVerifying AND digits.length >= 4 before appending |
| T-5-06 | Tampering | Double-tap triggering multiple verifyPin calls | mitigate | isVerifying state flag set to true before verifyPin call; handleDigit returns early if isVerifying is true; 300ms delay before reset on failure |
| T-5-SC | Tampering | npm/pip/cargo installs | accept | No new packages in this phase |
</threat_model>
