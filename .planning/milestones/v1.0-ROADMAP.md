# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-05-20
**Phases:** 1–7
**Total Plans:** 15

## Overview

Six planned phases + one gap-closure phase (Phase 7) took Math Tutor from a bare Vite+React scaffold to a fully offline-capable, iPad-optimized math tutor. Phases executed in strict dependency order: app shell and storage → curriculum data → narrated lesson player → practice engine → progress tracking + parent dashboard → PWA/offline capability → lesson catalog.

## Phases

### Phase 1: Foundation

**Goal**: A working app shell runs on iPad Safari with correct viewport handling, persistent local storage, and all tooling wired up — every subsequent phase builds on this without revisiting plumbing.
**Depends on**: Nothing (first phase)
**Requirements**: PLAT-01, PLAT-02, PLAT-03, PLAT-04
**Plans**: 2 plans

Plans:
- [x] 01-01-PLAN.md — Scaffold, Tailwind v4, Dexie singleton, React Router wiring, and Vitest test infrastructure
- [x] 01-02-PLAN.md — HomeScreen with Remy the Fox, styled stub screens, routing tests, and human verification checkpoint

**Details:**
- Vite 6 + React 19 + TypeScript 5 scaffold
- Tailwind v4 CSS-first configuration with iPad-safe tokens
- Dexie 4 singleton (MathTutorDB) with 3-table v1 schema

- iPad viewport (viewport-fit=cover, 100dvh, safe-area-inset)

- [x] 03-02-PLAN.md — LessonScreen, StepCard, tap-to-replay, narration playback

## Milestone Summary

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Web app (not native) | Faster to build, no App Store friction, still iPad-compatible | ✓ Good — no issues |
| Single device profile | Eliminates auth complexity, targets single-family iPad use | ✓ Good |
| Local storage only (Dexie) | No backend = simpler v1, parent data stays on device | ✓ Good |
| React Router 7 unified import | v7 uses `react-router` not `react-router-dom` | ✓ Good — documented early |
| 100dvh inline style (not Tailwind h-screen) | Tailwind resolves to 100vh; 100dvh required for iPad Safari | ✓ Good — all screens correct |
| Howler.js audio manager | Howler handles iOS Safari audio quirks cleanly | ✓ Good |
| Pre-recorded audio (stubs in v1) | Robot TTS kills tone; real audio deferred to production | — Deferred |
| useLessonAudio per-step Howl | One Howl instance per step, cleanup on unmount | ✓ Good |
| satisfies (not as) for curriculum.json | Compile-time type safety without runtime overhead | ✓ Good |
| Inline catalog JSX | Avoids new file/test scope for catalog-only change | ✓ Good |

**Issues Resolved:**
- iOS Safari audio unlock pattern established as load-bearing Phase 3 decision
- justReplayedRef guard prevents double-play on REPLAY dispatch
- act() wrapper required when calling Howl onend() in tests (React state update)
- useLiveQuery returns undefined for both loading and record-not-found — .then(r => r ?? null) disambiguates

**Issues Deferred:**
- iPad Safari verification: install from Safari, standalone mode, offline mode — user deferred
- Pre-recorded MP3 narration files: path stubs on disk (design decision D-04, production audio not yet recorded)
- VERIFICATION.md for phases 01, 03, 04, 06 — not generated before verify-work step was enforced

**Technical Debt Incurred:**
- 4 phases (01, 03, 04, 06) missing VERIFICATION.md — functional evidence via named tests only
- REQUIREMENTS.md traceability table stale for 16/24 requirements (checkboxes not updated)
- Phase 7 requirements (CAT-01-04) absent from REQUIREMENTS.md traceability (added post-freeze)
- Pre-existing TypeScript errors in phases 02–04 deferred (0 errors at milestone close)
